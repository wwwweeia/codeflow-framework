# 基于 Claude Code 的 Spec Coding 项目实战技术分享 — 大纲

## Context

公司推广 AI 数字人协作工作流，需要基于灵智 AI 项目的实践经验，向其他技术团队做技术分享，推动统一规范、提高开发效率。目标听众技术水平参差不齐，多数人对 AI Coding 停留在 Copilot 补全认知。老板和技术领导会旁听。

**分享目标（三层递进）**：
1. 让人"知道" — 布道，激发兴趣，消除偏见
2. 让人"会用" — 实操指南，能复制到自己项目
3. 让人"认同" — 推动公司层面采纳

**分享参数**：
- 时长：75 分钟（+ Q&A 自然延伸）
- 形式：PPT 讲解 + 现场演示 + Q&A
- 核心诉求：认知冲击 + 可复制性并重

**叙事策略**：案例驱动型 — 先"炸场"再拆解，先让人"看到"再让人"懂"

## 第一节：开场 — "同样的 AI，为什么效果差 10 倍？"（15 分钟）

**目标**：建立认知框架 — 问题不在工具，在认知

**内容结构**：

1. **认知冲击：输入阶梯效应**（6 分钟）
   - 开场直接抛出核心问题："同样的 Claude Code，有人一周交付一个完整平台，有人连一个 CRUD 都得改三遍——差距在哪？"
   - 用我们灵智 AI 项目的真实经历：初期没有规范时，AI 产出的代码风格各异、前后端对接混乱、每个模块一套规矩
   - 引入关键洞察：**输入质量和输出质量不是线性关系，是阶梯式的**
     - 输入从 60→80 分，输出可能从 40 直接跳到 85
     - 原因：输入模糊时 AI 要"猜"——猜架构、猜意图、猜标准，每猜一个都可能错，错误叠加
     - 输入清晰时不用猜，输出质变
   - 一句话总结：**瓶颈不在 AI 的能力，在你的思考质量**

2. **角色转变：你做决策，它做执行**（5 分钟）
   - 核心理念：**你是架构师，Claude Code 是你的工程团队**（不是比喻，是真实协作模式）
   - 引入"三层分工"框架：
     - 第一层（必须你做）：产品边界、架构决策、技术取舍、跨模块一致性
     - 第二层（AI 做你验收）：业务代码、接口、前端页面、测试用例、文档
     - 第三层（AI 全权处理）：格式化、样板代码、简单重构、启动脚本
   - 精力分配的转变：传统模式大量精力耗在低价值任务（修 bug、写样板、写文档），AI 接手后精力集中在架构决策和核心判断上——**这是复利效应**
   - 关键金句：**"AI 是放大器，不是发动机。你有方向它放大十倍，你没方向它放大的是零。"**
   - 铺垫：角色分工明确了，但怎么把决策传达给 AI？

3. **引出 SDD：规范 vs 好 Prompt**（4 分钟）
   - 类比：带团队时，口头说"做个用户系统" vs 给接口文档+命名规范+错误码定义——交付质量差一个档次
   - Claude Code 的记忆问题：上一轮对话给的规范，这一轮它完全不知道。**单次好 Prompt 只解决单次问题**
   - 关键区分：**写好 Prompt 是一次性的技巧，SDD 是贯穿项目生命周期的方法论**
   - 类比：SDD 和"写好 Prompt"的区别，就像"公司制度"和"口头交代"的区别——制度写下来，每次自动生效
   - 一句话预告：SDD 让 AI 永远在你的轨道上跑——接下来 25 分钟演示这套体系怎么运转

## 第二节：核心演示 — "从需求到代码"（25 分钟）

**目标**：视觉冲击 + 建立"原来如此"的体感

**演示案例**：灵智 AI 项目 — "新增预设问题管理功能"

**内容结构**：

1. **背景介绍**（1 分钟）
   - 灵智 AI 项目是什么
   - 展示需求："预设问题管理，支持 CRUD + 排序"
   - 强调：这不是 cherry-pick 的 demo，是日常开发的真实方式

2. **Intake 阶段**（3 分钟）— AI 不急于写代码，先问三个问题
   - 目标是什么？边界在哪？怎么验收？
   - 关键展示：AI 主动拒绝跳过需求澄清
   - **认知冲击点**：AI 也能"慢下来"

3. **Spec 生成 + 审批**（4 分钟）— PM Agent 输出需求文档
   - 展示 `01_requirement.md` 产出物
   - 关键展示：人工审批节点 —— 人做决策，AI 做执行
   - **认知冲击点**：人在回路中，不是放任 AI

4. **设计 + 并行开发**（12 分钟）— Arch → Dev/FE 并行
   - Architect Agent 输出技术设计（API 设计、DB Schema）
   - 工作流 C 亮点：Git Worktree 让前后端并行不冲突
   - 展示代码生成过程（后端 Java + 前端 Vue）
   - **认知冲击点**：AI 按工程规范走完整开发流程，不是写一段代码

5. **QA Review + 合并**（3 分钟）— 质量门控
   - QA Agent 独立审查，展示验收报告结论
   - **认知冲击点**：AI 还能帮 review，不只是写代码

6. **结果全景**（2 分钟）— 最终产出物
   - 后端 7 个文件 353 行 + 前端 4 个文件 615 行
   - Spec 文档 1059 行完整记录
   - 人工介入仅 5 个决策点
   - 总耗时：35-40 分钟

## 第三节：拆解 — "背后的设计秘密"（15 分钟）

**目标**：理解机制 + 看到设计深度

**内容结构**：

1. **SDD 三铁律**（2 分钟）
   - **No Spec, No Code** — 需求先行，未形成清晰 Spec 前不进入代码实现
   - **Spec is Truth** — Spec 是需求和实现的唯一真相源
   - **No Approval, No Execute** — 人工在回路，未得到明确批准不执行高风险操作
   - 一句话对比：Copilot 是"你问我答"，SDD 是"你审批我执行"

2. **架构设计：两层分离**（4 分钟）
   - 展示架构总览图：**编排层**（框架 `core/`）+ **执行层**（项目 `.claude/`）
   - 为什么分两层？框架统一标准，项目保留个性化——传统方案只能二选一
   - 关键设计决策：
     - 七角色定义放在编排层，所有项目共享同一套 Agent 行为
     - 项目特有规范（编码规范、业务词典、分支策略）放在执行层
     - 通过 MANIFEST 清单控制哪些文件被框架管理
   - 一句话：**框架管"怎么协作"，项目管"协作什么"**

3. **工作流路由：四种模式**（4 分钟）
   - 展示工作流路由图
   - 四种模式按复杂度递进：
     | 工作流 | 适用场景 | 角色链 |
     |--------|----------|--------|
     | Q0 轻量模式 | 单文件改动、小 bug 修复 | 直接编码 |
     | 工作流 A | API 设计、数据库变更、后端业务逻辑 | PM → Arch → Dev → QA |
     | 工作流 B | 页面开发、组件开发、前端交互 | PM → Arch → FE → QA |
     | 工作流 C | 新增业务实体、前后端联动 | PM → Arch → Dev + FE → QA |
   - 核心机制：Intake 阶段自动判断需求属于哪个工作流，然后按角色链调度
   - **认知冲击点**：不是人去选工作流，是 AI 根据需求特征自动路由

4. **Stub Marker + 双向工具链**（5 分钟）
   - Marker 机制（2 分钟）：一条分界线解决"统一 vs 个性化"矛盾
     - 上方框架管：`upgrade.sh` 自动更新
     - 下方项目管：升级时完整保留
     - 类比：像 Git 的 `.gitignore` — 一条线划清边界
   - 双向工具链（3 分钟）：
     - upgrade.sh：框架 → 项目（统一标准，一条命令升级）
     - harvest.sh：项目 → 框架（验证过的经验沉淀回 core/）
     - 三层验证闭环：demo 首发验证 → 真实项目验证 → 沉淀回框架
   - 核心信息：不是一次性工具，是持续演进的生态

## 第四节：文档站 — "回去怎么上手"（5 分钟）

**目标**：给出自助学习路径，降低采纳门槛

**内容结构**：

1. **文档站概览**（2 分钟）
   - 基于 VitePress 搭建，内网可访问
   - 按用户旅程组织的三条路径：
     - **入门指南**：5 分钟认识 SDD → 15 分钟快速入门 → 30 分钟端到端教程
     - **架构详述**：9 个章节完整参考，想深入了解机制时查阅
     - **参考手册**：7 个 Agent 定义、Skill 列表、命令手册、更新日志（自动生成，始终最新）

2. **接入方式**（2 分钟）
   - 新项目：`init-project.sh` 一键接入
   - 已有项目：参考文档站的接入指南，按检查清单逐步接入
   - 遇到问题：`doctor.sh` 环境诊断，一键检查依赖是否就绪
   - 我们团队提供接入指导和模板定制

3. **一句话总结**（1 分钟）
   - "AI 不会取代程序员，但会用 AI 的程序员会取代不会用的。Spec Coding 让你会用。文档站是你们的上手入口。"

## 第五节：Q&A（15 分钟）

**目标**：解答疑问，消除顾虑

**内容结构**：

1. **高频问题（预准备）**（9 分钟）— 每个问题 3 分钟

   **Q1：这套工作流会不会增加开发成本？AI 调用费用怎么算？**
   - 短回答：看场景。Q0 轻量模式几乎零额外成本，和平时用 Copilot 一样
   - 完整工作流（A/B/C）会多消耗 token，但省下的是返工和 review 时间
   - 灵智项目的实际数据：一个完整 Feature（工作流 C），AI token 成本约 X 元，但开发周期从 2-3 天缩短到半天
   - 关键视角：不要算"AI 多花了多少"，要算"返工少花了多少"

   **Q2：不是所有项目都适合这套规范吧？什么场景不适用？**
   - 不适用场景：纯探索性原型、一次性脚本、个人工具
   - 适用场景：有明确需求的中大型功能开发、团队协作项目、需要持续维护的业务系统
   - 核心判断标准：**这个需求的变更成本高不高？** 变更成本高的（多人协作、前后端联动），SDD 的收益就大

   **Q3：团队成员 AI Coding 水平参差不齐，怎么推广？**
   - 分层推进：先让 1-2 个人用 Q0 模式跑通，产出可见成果
   - 框架内置了 Intake 机制——AI 会主动引导使用者澄清需求，降低使用门槛
   - 文档站有完整教程和练习，新人可以自助学习
   - 关键：不要一步到位推全套，先让一部分人尝到甜头

2. **开放提问**（6 分钟）
   - 现场收集问题，自由回答
   - 备选话题（如果现场冷场）：学习曲线、与传统 Code Review 的关系、数据安全

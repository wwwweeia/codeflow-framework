# 如何生成一份高质量的 Domain Codemap

> 本文总结自实际生成过程中的经验沉淀。
> 目标读者：arch-agent（首次为某业务域生成 codemap 时阅读）。

---

## 一、生成前：明确边界，不要贪大

**最容易犯的错：不划边界就开始 Research。**

**正确做法：先写第 1 节"功能边界"，再开始读代码。**

问自己三个问题：
1. 这个域的核心操作是什么？（CRUD + 特有业务动作）
2. 哪些实体/功能**看起来相关但不属于这里**？（写进"不展开"列表）
3. 涉及哪些项目？（后端 / 哪个前端 App）

写完边界后，Research 范围就自然收敛了。

---

## 二、Research 顺序：从入口往里读，不要从实现往外猜

**推荐的阅读顺序**：

```
1. Controller（了解全部 REST 入口 + base path）
      ↓
2. Service 接口（了解有哪些方法）
      ↓
3. Service 实现的关键方法（重点看最复杂的那个，通常是 create/update）
      ↓
4. Mapper 接口（了解哪些是 XML 查询，哪些是 ORM 默认方法）
      ↓
5. Entity / VO / Query（了解字段结构和转换关系）
      ↓
6. 定时任务 / 外部集成（了解异步路径）
      ↓
7. 前端 Store → 前端页面（了解 API 调用约定和页面结构）
```

**不要一开始就读 XML 或 Service 的所有私有方法**，先建立全局视图，再按需下钻。

## 三、最容易漏掉的内容

### 3.1 多个 Controller 操作同一个 Service

不同 Controller（不同 base path）可能都注入了同一个 Service。

**检查方法**：搜索 Service 接口名被哪些 Controller `@Autowired`。

```bash
# 示例：搜索某 Service 的所有注入点
grep -r "XxxService" --include="*.java" -l
```

### 3.2 核心流程内的外部集成（不只是同步时才调）

外部调用不只在同步任务中出现，create/delete 主流程内也可能直接调外部系统，不只是同步任务才调。

**检查方法**：读 create / update / delete 方法的完整实现，不要只读方法签名。
重点看是否有 `thirdParty/`、`client`、`http`、`websocket`、`ssh` 相关调用。

### 3.3 私有方法中的隐式约定

枚举值的判断逻辑（如 `startsWith("group")`）、字段转换逻辑（如 `agentType` 存储值 vs 展示值）。

这类隐式约定**不在接口上暴露**，但影响所有上层逻辑。

**检查方法**：读 Service 实现的私有方法，关注 if/switch 的判断条件和字段转换逻辑。

### 3.4 内部私有关键方法（如 fillRelationData）

`fillRelationData` 在每次列表和详情返回时都执行，但它是私有方法，不在接口上。如果漏掉，后续加关联字段时完全不知道从哪里改。

**检查方法**：在 `convertToVO` 或 `buildVO` 类似的 VO 组装方法里，找所有被调用的私有方法，至少在 Service 层表格里记一行。

### 3.5 补偿逻辑和非原子操作

外部调用成功 → DB 保存失败 → 需要手动补偿删除外部资源。这是非原子的，补偿也可能失败，是真实的风险热点。

**检查方法**：阅读 `@Transactional` 方法时，特别关注方法内的**外部调用**（HTTP/SSH/MQ），分析"外部调用成功但 DB 失败"的场景下代码怎么处理。

## 四、各节填写要点

### 第 2 节：入口总览

**后端入口表格**多加一列"调用时机"（如：手动触发 / 定时任务 / 创建主流程），特别是外部集成点，调用时机往往不直观。

**前端入口**：先列 Store Action，再列页面路由。因为 Action 是 FE 和后端 API 的桥梁，比页面更稳定，更容易与后端对齐。

### 第 3 节：主流程图

**只画主链路，不画全部分支**。控制在 15 个节点以内。

如果业务有复杂创建流程（如多种路径 + 外部调用 + 补偿），**单独画一张创建流程图**，不要把所有流程挤在一张图里。

### 第 4 节：后端模块分层

Controller 层：**写清楚 base path**，不要只写类名。多个 Controller 操作同一个 Service 时要分开列。

Service 层：**私有但关键的方法也要记一行**，标注"（私有）"和调用场景。

外部集成点：**增加"调用时机"列**，区分是主流程调用还是异步同步任务调用。

### 第 6 节：关键数据结构

字段备注里写清楚**枚举值**（如 `status`：active/inactive），不要只写字段名。枚举值是隐式约定，最容易被新人搞错。

VO 和 Entity 的字段若有转换（如存储值 vs 展示值），在表格备注里标注，或者在风险热点里单独说明。

### 第 7 节：影响分析

这节是**动态工作区**，不是静态内容。
- Feature 开发前：arch-agent 填入本次具体改动点
- Feature QA PASS 后：arch-agent 清空，恢复为占位状态

占位状态格式固定写：`_(Feature 开发前由 arch-agent 填入)_`，便于识别。

### 第 8 节：风险热点

**每条风险必须写清楚**：
1. 风险的本质是什么（不是笼统说"复杂"）
2. 在哪个文件/方法里（附行号）
3. 踩坑场景（什么操作会触发这个风险）

---

## 五、完成后自检清单

生成完毕后，对照以下清单检查：

- [ ] 第 1 节"不展开"列表写了吗？（边界不清就是不完整）
- [ ] 是否搜索过 Service 接口被几个 Controller 注入？（防止遗漏入口）
- [ ] 是否读了 create / update / delete 方法的完整实现？（不只是方法签名）
- [ ] 外部集成点是否加了"调用时机"列？（区分主流程 vs 同步任务）
- [ ] 是否有私有但关键的方法没有记录？（如 fillRelationData、convertToVO）
- [ ] Entity / VO 的字段转换关系是否记录？（特别是枚举值和隐式映射）
- [ ] `@Transactional` 方法内有外部调用时，是否分析了补偿逻辑？
- [ ] 风险热点每条是否写了文件路径和行号？
- [ ] 第 9 节切入点是否给出了具体行号？（不能只写类名）
- [ ] 总行数是否在 200 行以内？（超出说明边界划太宽，考虑拆域）

## 六、更新时机（Feature 完成后）

QA PASS、代码合并后，主会话提示 arch-agent 更新对应 domain codemap：

**需要更新的节**：
- 第 4 节：新增/修改的 Controller 方法、Service 方法、Mapper
- 第 5 节：新增/修改的页面、Store、组件
- 第 6 节：新增/修改的表字段、枚举值
- 第 8 节：新发现的风险或已消除的风险
- 第 7 节：**清空**，恢复占位状态

**不需要更新的节**：第 1、2、3、9 节通常保持稳定，除非有重大架构调整。

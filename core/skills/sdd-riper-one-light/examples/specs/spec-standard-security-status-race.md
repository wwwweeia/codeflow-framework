# Spec: 协作资源安全状态竞争修复（公开示例）

## Goal
- 要解决什么问题：
  某类协作资源在异步安全检测过程中出现状态竞争，先写入了偏乐观的中间状态，随后又被后续回调修正为更严格的最终状态，导致客户端可能短暂读取到错误结果。
- 验收结果：
  同一资源在该场景下不再先暴露错误中间状态，再被后续异步流程纠正；对外只呈现最终可信状态。

## Scope
- In:
  安全状态管理模块
  状态回写处理模块
  资源元数据更新模块
- Out:
  客户端改造
  下载链路策略改造
  与当前问题无关的规则扩展

## Facts / Constraints
- 已确认事实：
  同一资源在一次问题样本中经历了“早期乐观状态”与“后续严格状态”的冲突回写。
  最终结果是严格状态正确，但问题在于中间过程对外暴露了错误信号。
  当前系统同时存在旧兼容链路与新检测链路，两者叠加后形成状态竞争。
  同一次状态更新存在重复写回，放大了问题表现。
- 技术/业务约束：
  不改变当前对外状态定义。
  不粗暴删除历史兼容逻辑，避免影响仍依赖旧链路的场景。
  代码修改前需先获得明确批准。
- 已知风险：
  如果只去掉重复写回，但不处理状态来源冲突，问题仍会存在。
  如果直接删除兼容分支，可能影响历史路径稳定性。

- [ ] 是否需要增加状态机保护，避免更严格状态被后续降级覆盖。

## Restated Understanding
- 我理解当前任务是：
  修复一个典型的异步状态竞争问题，避免资源对外短暂暴露错误安全状态。
- 当前核心目标是：
  让状态更新只沿着可信方向收敛，并消除重复写回。
- 当前边界是：
  只处理服务端上传阶段的状态竞争，不改客户端，不改其他链路策略。
- 暂不处理：
  状态定义扩展、客户端消费逻辑调整、无关模块重构。

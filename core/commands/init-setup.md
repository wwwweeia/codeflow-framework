逐步驱动项目初始化清单，自动检测项目信息并引导用户完成配置。

## 使用方式

```
/init-setup              # 从第一个未完成任务继续
/init-setup --status     # 查看当前进度摘要
/init-setup --skip T3    # 跳过指定任务（标记为 skipped）
/init-setup T2           # 跳转到指定任务执行
```

## 执行步骤

### 第零步：读取清单

读取 `.claude/setup-checklist.md`。

- 若文件不存在：提示用户先执行 `sh <框架路径>/templates/init-project.sh . "<项目名>"` 完成初始化
- 若文件存在：解析全部任务块，提取 `status`、`required`、`level`、`files`、`auto-detect`、`ask-user`、`verify` 字段

### 第一步：参数处理

根据 `$ARGUMENTS` 分支：

**`--status`**：输出进度摘要表格后停止，不执行任何任务：
```
## 初始化进度 — <项目名>

| 任务 | 状态 | 必需 |
|------|------|------|
| T1 填充 CLAUDE.md          | ✅ done (2026-04-23 15:30) | 是 |
| T2 填充业务词典             | ⏳ pending                  | 是 |
| T3 配置 MCP 集成            | ⏸ skipped                  | 否 |
| ...                        |                             |    |

进度：1/8 完成，1/8 跳过，6/8 待完成
下一个任务：T2 填充业务词典
```

**`--skip T<N>`**：将指定任务的 `status: pending` 改为 `status: skipped`，输出确认后停止

**`T<N>`（任务 ID）**：跳转到指定任务执行（无论其状态如何）

**无参数**：找到第一个 `status: pending` 的任务，按下方"执行任务"流程处理

### 第二步：执行任务

对当前任务，按以下顺序执行：

#### 2a. 宣告任务
输出：
```
---
▶ 正在执行：T<N> — <任务标题>
```

#### 2b. 自动检测
按任务的 `auto-detect` 字段描述，扫描项目：
- 读取构建文件（package.json、pom.xml、build.gradle、requirements.txt）
- 扫描源码目录结构
- 读取现有配置文件（.eslintrc、checkstyle.xml 等）
- 检测框架路径和版本

将检测结果组织成清晰的摘要，不要输出原始文件内容。

#### 2c. 呈现草稿
基于检测结果，为用户生成当前任务的**填写草稿**：
- T1：生成 CLAUDE.md 填充建议（技术栈、架构描述等）
- T2：生成业务词典初始条目表格
- T4：生成检测到的代码风格摘要
- T6：生成子项目上下文的骨架内容

明确标出哪些是自动检测到的、哪些需要用户补充。

#### 2d. 询问用户
按任务的 `ask-user` 字段，逐项询问需要用户提供的信息。

**T3（MCP 配置）特殊处理**：凭据信息（用户名/密码）逐项询问，不要一次性全部问。处理完后提醒用户确认 `.gitignore` 已包含 `.mcp.json`。

**T5（升级）特殊处理**：先用 `--dry-run --diff` 预览变更，让用户确认后再执行真正的升级。

**T7（Spec 目录）特殊处理**：仅说明 Spec 工作流，告知用户首次创建 Feature 时 PM Agent 会自动建立，询问是否已了解，无需用户填写任何内容。

**T8（E2E）特殊处理**：询问是否需要 E2E 基础设施；若需要，执行 `sh <框架路径>/templates/e2e/init-e2e.sh <项目路径> <项目名>`；若不需要，标记为 skipped。

#### 2e. 写入文件
将草稿 + 用户输入合并，写入任务 `files` 字段指定的文件：
- 对于有 marker 的文件（domain-ontology、coding rules）：内容写在 marker **下方**，不动 marker 上方的框架内容
- 对于 CLAUDE.md：替换占位符 `[请填写...]`，保留已有结构
- 对于 .mcp.json：用 JSON 更新，保持格式整洁

#### 2f. 验证
按任务的 `verify` 字段检查结果。验证通过后继续；不通过则告知用户问题并询问是否重试。

#### 2g. 更新状态
将 `.claude/setup-checklist.md` 中对应任务的状态行从：
```
- status: pending
```
改为：
```
- status: done (YYYY-MM-DDTHH:MM:SS)
```

#### 2h. 询问继续
输出当前进度，询问：「继续下一个任务（T<N+1>）？还是先到这里？」

用户确认后，自动进入下一个 `pending` 任务；用户说「先到这里」则停止并输出进度摘要。

### 第三步：全部完成

当所有任务均为 `done` 或 `skipped` 时，输出：

```
🎉 初始化配置全部完成！

已完成：N 个任务
已跳过：M 个任务

项目已就绪，可以开始第一个 Feature：描述你的需求，主对话将启动 Intake 三问。
```

## 约束

- 每次只处理**一个**任务，完成后明确询问是否继续，不要自动连续执行多个任务
- 写入文件前必须向用户展示将要写入的内容，不要静默覆盖
- T3（MCP 配置）涉及密码等敏感信息，不在对话中回显完整密码
- 只修改任务 `files` 字段明确指定的文件，不要修改其他文件
- 状态更新（2g）必须在验证通过后才执行

用户的原始输入（如果有）：$ARGUMENTS
